Phase 1 — Anchor program
H 1 → H 5
P1 owns ⌄
P1 Rust
H1–2: Define DareAccount struct + DareStatus enum. Verify #[derive(InitSpace)] compiles. Implement create_dare with PDA seeds [b"dare", creator, dare_hash] and system_program::transfer CPI.
PDA vault trick: the DareAccount itself IS the vault — lamports transfer directly into the PDA account. No separate vault account needed. Simpler, fewer accounts, less surface area to break.
P1 Rust
H2–4: Implement accept_dare, approve_dare, and reclaim. Each must check Clock::get()?.unix_timestamp vs expires_at. approve_dare emits ApprovalEvent and returns SOL to creator. reclaim uses close = creator to return rent.
If Rust compiler is fighting you at H3: strip reclaim, deploy 3-instruction program. Three instructions still qualifies. Add reclaim after the H5 devnet milestone.
P1 Rust
H4–5: Build → fix all compiler errors → deploy to devnet via Playground → copy Program ID → paste into .env everywhere → verify on Solana Explorer.
Common error: has_one constraint not matching field name. Check that has_one = creator corresponds to a field literally named creator in the account struct.
P4 Integ
H4: Export IDL from Playground → save as /app/idl/soldare.json → wire into anchor-client.ts → smoke-test getDarePDA() locally.
IDL export: Playground sidebar → IDL tab → copy JSON. This is the P1→P4 handoff. Everything downstream depends on this file being correct.
Milestone A (H5): Program deployed to devnet. Program ID confirmed on Solana Explorer. First create_dare tx visible on-chain. Qualification requirement #1 and #2 done.
Phase 2 — UI + x402 backend
H 5 → H 9
P2 + P3 ⌄
P2 Front
H5–7: Build the three core pages. (1) /create — dare text + bounty + expiry form, wallet connect, calls createDare(), inserts into Supabase, shows shareable link. (2) /d/[id] — public dare view, accept button. (3) /d/[id]/prove — proof upload form.
Shareable link design: soldare.xyz/d/{supabase_uuid}. No wallet needed to VIEW the page — that's the x402 UX win. Show countdown timer, bounty in USDC (not SOL), dare text.
P3 x402
H5–7: Wire @x402/next middleware in middleware.ts. Build /api/dare/payout route — accepts dare_id + recipient + bounty_usdc_cents → sends USDC from treasury wallet → updates Supabase. Build /api/dare/create server action that inserts dare into Supabase.
Treasury wallet must hold devnet USDC before H6. Fund it from the Circle faucet. The treasury is the payer for recipient's ATA creation — this is why the recipient never needs SOL (gasless UX).
P3 x402
H7–8: Set up Helius webhook — devnet, webhook URL = Vercel preview URL, account = Program ID, tx type = any. Implement /api/webhooks/helius to catch ApprovalEvent and trigger the payout route.
Fallback if Helius webhook is flaky: after creator signs approve_dare tx, the frontend calls /api/dare/payout directly with dare_id + recipient. Less elegant, still works, still demoes perfectly.
P4 Integ
H6–8: Build useSolDare React hook — wraps all Anchor calls (createDare, acceptDare, approveDare, reclaimDare, fetchDareAccount). P2's buttons call this hook, not raw Anchor directly. Keeps integration surface clean.
Critical: P1 freezes the program API at H5 — no instruction signature changes after that. P4 can trust the IDL is stable from H5 onward.
Milestone B (H8): Create dare form → Supabase insert → on-chain create_dare tx → shareable link generated. All three working together. Not yet E2E, but the create path is done.

Integration points — where things break
Anchor IDL → frontend (H5)
P1 exports IDL from Playground. P4 drops into /app/idl/soldare.json. P1 freezes the instruction API at H5 — no signature changes after this. Biggest risk: IDL mismatch if P1 renames a field. Fix: P4 re-exports and tests immediately after any P1 change.
Approval event → x402 payout (H8)
Helius webhook catches ApprovalEvent → calls payout route → USDC sent. Test in isolation before wiring to UI. If Helius is flaky: creator's browser calls payout route directly after approve_dare tx confirms. Fallback is one fetch() call.
On-chain ↔ Supabase sync (H7)
On-chain DareAccount is truth for money. Supabase is truth for content (dare text, proof URL). P4 owns the sync: create_dare tx confirms → P4's hook updates Supabase status. Never derive money state from Supabase alone.
x402 facilitator config (H0)
P3 registers with Coinbase CDP at H0. Free tier covers the demo. If CDP has a waitlist, direct SPL transfer is the fallback — same payout route, swap the implementation. Never let facilitator setup block Phase 2.
Risk register
P1 Rust stuck at H3+ — strip reclaim, deploy 3-instruction program. Still qualifies. P4 handles refund client-side if needed. Don't let one instruction sink the whole program.
x402 facilitator issues at H8+ — switch to direct @solana/spl-token transfer in the payout route. Same UX for recipient (gasless, instant). Mention x402 in README as the intended design. Still impressive.
Devnet airdrop rate limits — P1 requests 2 SOL at H0. P3 gets devnet USDC from Circle faucet at H0. Fund the treasury wallet at H0. Do not wait until H6.
IDL mismatch after P1 changes — P1 API freeze at H5. Any changes after that are internal-only (no instruction/account renames). P4 re-exports IDL immediately if P1 must change something before freeze.
Demo SOL/USDC runs out during judging — P4 keeps a separate funded demo wallet. Never the dev wallet. Check balances at H14:30. Top up if below $5 USDC.
