Phase 2 — Core build (parallel sprints)
H 1 → H 6
parallel lanes ⌄
P1 Rust
Write all 4 instructions in sequence:
H1–2: create_dare — system_program CPI transfers lamports from creator into PDA vault, sets expiry via Clock::get()
H2–3: accept_dare — checks signer == recipient, flips status to Accepted
H3–4: approve_dare — checks signer == creator, status == Done, emits ApprovalEvent
H4–5: reclaim — checks Clock::get()?.unix_timestamp > expires_at, returns SOL to creator

Integration points — the 3 moments where things can break
Anchor IDL → frontend (H6)
P1 exports IDL from Playground. P4 drops it into /app/idl/soldare.json. P2's buttons call P4's client functions. This must happen before devnet testing. Biggest risk: IDL mismatch after P1 changes instructions.
On-chain event → x402 payout (H8)
Helius webhook fires on ApprovalEvent → hits P3's payout route → x402 sends USDC. Test this in isolation first. If Helius webhook is flaky, fallback: creator manually calls payout route after signing.
On-chain + Supabase sync (H7)
Dare state lives both on-chain (DareAccount PDA) and in Supabase (dare text, proof URL). P4 owns this sync: on-chain is truth for money, Supabase is truth for content. Never mix them up.
x402 facilitator config (H6)
P3 must register with Coinbase CDP for the facilitator API key before the hack starts. Free tier: 1000 tx/month — more than enough. If facilitator is down, fallback: direct SPL token transfer via @solana/spl-token.
Risk register
P1 Rust stuck on compiler (H3+) — fallback: strip reclaim instruction, deploy 3-instruction program. Still qualifies. P4 handles USDC return client-side.
x402 facilitator issues (H8+) — fallback: remove x402, do direct SPL USDC transfer in frontend using @solana/spl-token. Less elegant but works. Still mention x402 in README as the intended design.
Devnet airdrop rate limits — P1 requests 2 SOL at H0 on Playground. P3 gets devnet USDC from the Circle faucet at H0. Do not wait until H6.
IDL mismatch after changes — P1 freezes the program API at H5, no instruction signature changes after that. Any fixes are internal only.
Demo SOL runs out — P4 keeps a separate funded devnet wallet for the demo. Never the development wallet.
